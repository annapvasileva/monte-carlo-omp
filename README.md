# Вычисление объёма фигуры методом Монте-Карло

## Инструментарий

*Стандарт языка*: С++20\
*Компилятор*: g++\
*Версия OpenMP*: OpenMP 2.0\
*Построение графиков*: Python3\
$\color{pink}{\textsf{Процессор}}$: AMD Ryzen 5 5500U with Radeon Graphics

## Что реализовано

Программа, вычисляющая объем фигуры (конкретная фигура определяется имплементацией функций в файле `hit.cpp`) методом Монте-Карло с использованием многопоточности и записывающая его в выходной файл.

# Описание

## Реализация

### Последовательное решение

Для начала рассмотрим решение задачи без использования многопоточности, а затем будем его модифицировать.

Метод Монте-Карло заключается в расчёте определенных характеристик объекта на основе множественных вычислений для сгенерированных для него случайных величин.

В нашем случае мы упростим объём данного тела до объёма куба, в котором это тело полностью помещается (оценим сверху), и будем приближать эти объемы. Делать мы это будем следующим образом: мы случайно выберем в кубе $N_1$ точек и посчитаем количество тех, которые содержатся внутри данного тела (Df: $N_2$). Тогда при стремлении $N_1$ к $\infty$ отношение $\frac{N_2}{N_1}$ будет стремиться к отношению объема данного тела к объему куба. Объем куба вычислить просто, а потому, зная его, мы сможем найти объем искомого тела, домножив отношение $\frac{N_2}{N_1}$ на объем куба.

Таким образом, [программа](https://github.com/annapvasileva/monte-carlo-omp/blob/980a6a1261c981fcb951cc6ab42926a32cb391ba/main.cpp#L68) в цикле генерирует $N_1$ точек и суммирует количество попадающих в данную фигуру. В качестве ответа она выводит произведение отношения $\frac{N_2}{N_1}$ и объёма куба, полностью содержащего в себе данное тело.

### Многопоточное решение

Заметим, что предыдущее решение имеет в своей основе цикл, состоящий из множества одинаковых итераций, причём он составляет основное время работы программы. А потому его выполнение выгодно разделить на потоки.

**Теория**\
Для распараллеливания программ на языках C/C++ (и Fortran) существует стандарт OpenMP. Данный стандарт содержит описания директив компилятора, функций и переменных окружения.

OpenMP использует *fork-join* модель выполнения параллельных программ, то есть исполнение "разветвляется" в определенных точках программы на несколько потоков, чтобы впоследствии "слиться" в какой-либо точке и возобновить последовательное выполнение.

Таким образом, программа, помимо обычных последовательных блоков (*serial regions*), содержит блоки с параллельным исполнением (*parallel regions*). Каждый parallel region привязан к *parallel construct* (*construct* – структурный блок с предшествующей ему директивой OpenMP: они начинаются с `#pragma omp` и определяют дальнейшее поведение программы).

Программа, написанная с использованием OpenMP, начинает свое выполнение с единственного потока – так называемого *master thread*'а. Master thread продолжает последовательное выполнение до тех пор, пока не встретит первый parallel construct. В этот момент master thread создает группу потоков (*team of threads*) и становится главным потоком группы (*master of the team*).

Для того, чтобы добиться синхронизации, существуют барьеры (*barriers*) – точки, которых перед продолжением выполнения должны достичь все потоки группы.

**Практика**:\
[В нашем случае](https://github.com/skkv-itmo-comp-arch/se-comp-arch24-omp-annapvasileva/blob/980a6a1261c981fcb951cc6ab42926a32cb391ba/main.cpp#L103) важно создать parallel region с заданным количеством потоков. Для этого существует директива `parallel`, имеющая синтаксис:

```cpp
#pragma omp parallel [clause[ [, ]clause] ...]
    structured-block
```

Она сообщает о начале parallel region. В качестве clause необходимо указать `num_threads(число)`, а также перед этим отключить динамическую регулировку количества потоков при помощи функции `omp_set_dynamic` (иначе количество потоков, используемых для выполнения последующих parallel regions, может быть автоматически скорректировано средой выполнения, причем значение, указанное в `num_threads`, будет восприниматься как максимальное количество потоков).

OpenMP хорошо умеет оптимизировать именно циклы `for`. Для них существует отдельный construct:

```cpp
#pragma omp for [clause[[,] clause] ... ]
    for-loop
```

При выполнении итерации цикла распределяются по уже существующим потокам, созданным в parallel construct.

`schedule` ([см. код](https://github.com/skkv-itmo-comp-arch/se-comp-arch24-omp-annapvasileva/blob/980a6a1261c981fcb951cc6ab42926a32cb391ba/main.cpp#L114)) определяет, каким образом итерации цикла `for` будут распределяться по потокам группы. Параметр `chunk_size` отвечает за размер чанков, т.е. наборов итераций, распределяемых по потокам. Если он отсутсвтует, то, в случае static, все итерации равномерно разделяются по имеющимся потокам, а в случае dynamic – `chunk_size` полагается равным 1. Параметр *kind*, может принимать следующие значения:

- static (итерации распределяются по принципу *round-robin*: назначения происходят по кругу вне зависимости от того, свободен ли поток, распределение происходит перед выполнением);
- dynamic (следующая порция итераций назначается незанятому потоку, распределение происходит во время выполнения);
- guided (размер чанков уменьшается от `chunk_size`);
- runtime (решение о планировании откладывается до времени выполнения: тип schedule и размер чанков могут быть определены в момент выполнения).

Теперь программа создает переменную `cur_thread_hits` (она имеет тип `private`, то есть каждый поток оперирует собственной копией данной переменной), складывает в нее результат суммирования, а в самом конце добавляет полученное значение к общей переменной `hits`.

Во избежание *race condition* инструкция

```cpp
hits += cur_thread_hits;
```

имеет предшествующую ей директиву `#pragma omp atomic`, которая говорит о том, что последующая инструкция (ровно одна) должна выполняться лишь одним потоком в единицу времени, и никогда – несколькими одновременно. Из-за этого инструкция получается довольно затратной (необходимо отслеживать количество потоков, пытающихся выполнить инструкцию, а также приостанавливать выполнение некоторых из них), а потому мы пытаемся сократить ее использование и обращаемся к ней только в конце выполнения подсчетов каждым из потоков.

После всех вычислений ответ вычисляется аналогично последовательной реализации.

$\color{red}{\textsf{Важное замечание}}$: так как программа теперь использует несколько потоков, может возникнуть race condition при одновременном обращении разных потоков к shared переменным. Подобное может случиться и с генератором случайных чисел. А потому нам необходимо, чтобы он был thread-safe. Таким образом, каждый поток в программе создает свой генератор (`pcg32_fast`), который используется локальными объектами distribution для генерации точек. Выбор генератора также обусловлен скоростью его работы, так как бóльшую часть работы программы занимает как раз генерация случайных чисел.

### Постоянный парметр schedule без `chunk_size`

**Static**:\
![first graphic](/graphics/graphic1.png)

Рассмотрим график зависимости времени от количества потоков при параметре static.

Изначально стоит оговориться, что значения, которые в нем представлены - усреднены по 5 запускам программы с указанными параметрами в попытке минимизировать влияние загруженности процессора на результаты работы программы. Несмотря на небольшие изменения в значениях при запуске в разные временные промежутки, вид графика остается неизменным, а потому по нему можно наблюдать определенные закономерности.

При единственном потоке время работы, очевидно, является самым большим. Далее, по мере увеличения количества потоков наблюдается резкий спад (распределение задач по нескольким исполнителям способствует увеличению производительности), который в определенный момент времени прекращается – дальнейшее увеличение числа потоков не улучшает производительность. Причина такого явления в том, что процессор имеет определнный предел по количеству одновременно запущенных потоков, а именно – количество ядер (как физических, так и логических: например, технология гипертрединга позовляет операционной системе одно физическое ядро рассматривать как два отдельных логических).

Таким образом, после определнного значения не имеет смысла увеличивать количество потоков, так как это не приведет к улучшению производительности, но только усложнит распределение (а значит, замедлит работу программы). Кроме того, операции создания и уничтожения потоков являются довольно затратными с точки зрения ресурсов.

**Dynamic**:\
$\color{red}{\textsf{Примечание}}$: исследование проводилось при количестве проб, равном $10^7$ с флагом компиляции -O2, чтобы сделать расчеты быстрее. Было замечено, что при количестве проб – $10^8$ время работы также приблизительно возрастало в 10 раз, но закономерности не менялись.\
![Second graphic](/graphics/graphic2.png)

Для параметра dynamic наблюдается иное поведение:

1. Время работы в общем значительно больше по сравнению с такими же условиями типа static. Это можно объяснить дополнительными расходами при распределении задач по потокам: определение свободного потока во время исполнения замедляет работу. Кроме того, так как параметр `chunk_size` не был указан, по умолчанию он равен 1, то есть каждую новую итерацию процессору приходится назначать какому-либо потоку. Режим dynamic стоит использовать в том случае, когда итерации не одинаковы по времени выполнения. В нашем же случае они все являются равноправными.

2. График возрастает до определенного момента, после чего наблюдается плато. Думаю, это происходит потому, что затраты на распределение итераций по потокам при увеличении их числа в режиме dynamic (а они происходят в runtime, причем распределяются по одной итерации, ввиду отсутствия параметра `chunk_size`) оказывают слишком большое влияние на производительность. То есть чем больше потоков, тем сложнее распределение по ним, учитывая, что каждую новую итерацию необходимо распределить.

Рост времени прекращается в определенной точке – это следствие того же факта, который объяснял прекращение убыли времени на предыдущем графике: количество одновременно работающих потоков на процессоре ограничено числом ядер, а потому их увеличение за пределы этого числа мало сказывается на ухудшении производительности.

Заметим, что они имеют общий вид: скачок наблюдается на малом `chunk_size` ввиду того, что происходит очень частое завершение выполнения каждым потоком своей "порции" итераций и назначение ему следующей. Повышаются затраты на синхронизацию.

Далее значения хоть идут скачками, но в целом сильно друг от друга не отличаются. Когда количество итераций не делится нацело на количество потоков, каким-либо потокам будет выдано бóльшее количество итераций, чем остальным. От остатка от деления будет зависеть время работы.

В определенный момент время начнет снова расти:

![more iterations, static](/graphics/graphic3_3.png)

Это объясняется неравномерной нагрузкой имеющихся потоков: в какой-то момент "порций" итераций остается меньше, чем существует потоков, реально способных работать параллельно, а потому увеличение `chunk_size` становится равносильным уменьшению числа потоков: какие-то из них простаивают, в то время как остальные занимаются вычислениями.

Заметим, что для разных значений количества потоков минимумы графика располагаются в разных точках, а потому подобрать универсальный `chunk_size` вряд ли удастся. По этой причине при изменяющемся количестве потоков удобнее использовать в качестве параметра `schedule` только `static`, без `chunk_size`.

### Оптимальная конфигурация

Таким образом, можем сделать вывод, что наилучшим параметром `schedule` из `static` и `dynamic` точно будет `static`, ввиду того, что все итерации одинаковы и нет необходимости распределять их во время исполнения каким-либо сложным образом.

Но стоит ли устанавливать определенный `chunk_size`? В спецификации говорится:

> When no *chunk_size* is specified, the iteration
space is divided into chunks that are approximately equal in size, with one
chunk assigned to each thread.

А также:

> For a `static` schedule, the default chunk size is set to the loop
iteration space divided by the number of threads applied to the loop.

То есть какому-либо потоку (если количество итераций нацело не делится на количество потоков) всегда достается на остаток от деления количества итераций на количество потоков больше, чем остальным. Так как потоков обычно немного, то не имеет смысла задавать определенный `chunk_size`, так как эта погрешность сильно не повлияет на вермя.
